# OAuth2/OIDC における GET と POST の使い分けとセキュリティ設計

OAuth2/OIDC のフローにおいて、なぜ `/authorize` は GET (URLパラメータ) を使い、`/token` は POST (Body) を使うのか。その理由は、Web の基本的な仕組みと歴史的経緯に深く根ざしています。

## 1. /authorize エンドポイント (GET)

### 役割

ユーザーを IdP (Keycloak, Google 等) のログイン画面へ連れて行くこと。

### なぜ GET なのか？

これは「ブラウザのリダイレクト」を利用するためです。
HTTP の仕様上、リダイレクト (302/303/307) は `Location` ヘッダに遷移先の URL を指定する仕組みであり、**Body を含めることができません**。

```http
HTTP/1.1 302 Found
Location: https://idp.example.com/auth?client_id=...&redirect_uri=...
```

クライアントアプリはブラウザをこの URL に飛ばすしかないため、必然的にパラメータは URL クエリ文字列に含めることになります。

### セキュリティリスク

URL に情報が含まれるということは、以下のようなリスクを伴います。

- ブラウザの履歴に残る
- Referer ヘッダとして外部サイトに送信される可能性がある
- プロキシやサーバーのアクセスログに残る
- ブラウザ拡張機能から読み取れる

### PKCE の必要性

この「URL に情報を載せざるを得ない」という Web の構造上の制約により、認可コード (code) が漏洩するリスクが生まれます。これを補うために **PKCE (Proof Key for Code Exchange)** が導入されました。
事前に `code_challenge` を送っておき、後で `code_verifier` を送ることで、URL 経由でコードを盗んだ攻撃者がトークンを取得できないようにします。

## 2. /token エンドポイント (POST)

### 役割

取得した認可コード (code) をアクセストークン等と交換すること。

### なぜ POST なのか？

これはブラウザのリダイレクトではなく、クライアントアプリから IdP への**直接的な API コール**だからです。
ユーザーの操作（画面遷移）を伴わないため、HTTP POST メソッドを使用でき、データを Body に格納できます。

### 安全性の担保

- **Body の使用**: URL ではないため、ブラウザ履歴やログに残りません。
- **TLS (HTTPS)**: 通信経路は暗号化されており、Body の中身はトンネルの中を通ります。

### 結論

`/token` の安全性は **HTTPS (TLS)** に依存しています。TLS が破られない限り、Body に含まれる `code_verifier` や `client_secret` は安全です。

## まとめ

| エンドポイント | 通信方式        | 理由                                       | セキュリティ                         |
| :------------- | :-------------- | :----------------------------------------- | :----------------------------------- |
| **/authorize** | **GET** (URL)   | ブラウザリダイレクトのため Body が使えない | URL 露出リスクあり → **PKCE で保護** |
| **/token**     | **POST** (Body) | API コールのため Body が使える             | **HTTPS (TLS)** で保護               |

OAuth2 は「現実世界の Web の挙動（リダイレクトの制約）」に合わせて設計されており、その制約の中で安全性を確保するために PKCE などの仕組みが組み合わされています。
